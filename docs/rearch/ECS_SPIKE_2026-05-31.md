# ECS Spike (combatant targeting) — promote/defer decision

Last verified: 2026-05-31

Branch: `task/phase4b-ecs-spike`
Status: Decision memo — corroborates the [E1 evaluation](E1-ecs-evaluation.md) (2026-04-16, projectiles). No human go/no-go is unblocked by this memo; it sharpens the existing DEFER with a second subsystem.

## TL;DR

**DEFER.** A native bitECS 0.4 struct-of-arrays (SoA) port of the combatant
nearest-enemy targeting scan is **bit-identical** to the current OOP path
(parity 10/10) but **not faster** — it measured **1.00x–1.09x** (i.e. ECS at
best tied, more often a few percent slower) at every tested scale, including
1000 (1.06x) and 2000 (1.00x) entities. The `docs/REARCHITECTURE.md` § E1
decision rule (promote only on **≥3x at 1000+ entities AND bounded**) returns
"don't" unambiguously. This independently confirms the 2026-04-16 E1 memo,
which tested a different subsystem (projectile physics) and got ~0.9x.

## 1. Question (same as E1)

Should combatants migrate from the object-graph model (`Map<string, Combatant>`
in `src/systems/combat/CombatantSystem.ts`) to a `bitECS` entity-component-
system model to unlock the 3,000-agent combat target?

The 2026-04-16 E1 memo answered "defer" using **projectiles** as the slice,
explicitly noting (E1 § 5) that a real *combatant* slice was the sharper future
test. This spike runs exactly that combatant slice.

## 2. What was mirrored

The representative read-heavy hot loop chosen is **nearest-enemy targeting** —
the brute-force (no-spatial-grid) branch of `AITargetAcquisition.findNearestEnemy`
(`src/systems/combat/ai/AITargetAcquisition.ts`). For each live combatant it
scans every other live combatant, skips self and allies, keeps enemies inside
visual range, and selects the nearest by squared distance. This is the combat
hot path the 3,000-NPC vision stresses hardest: O(N²) distance work, purely
read-over-positions, so it mirrors a SoA layout exactly and is the *most
favorable* case for ECS cache behavior (dense linear streaming of position
columns).

Self-contained spike modules (imported ONLY by the parity test + the benchmark
script — never by any production module; verified, see § 6):

- `src/ecs/CombatantComponents.ts` — SoA component columns (Float32 position/
  velocity/health, Int32 state/faction/target/squad) + faction/alliance ordinals.
- `src/ecs/CombatantEcsWorld.ts` — bitECS 0.4 world builder + `ecsAssignNearestEnemy`.
- `src/ecs/CombatantOopReference.ts` — Array-of-Structs baseline (`Vec3F32`-backed
  records) + `oopAssignNearestEnemy`, the in-memory shape of the live `Combatant`.
- `src/ecs/CombatantSpikeSeed.ts` — deterministic integer-LCG seed generator
  (no `Math.random`, no clock).
- `src/ecs/EcsParity.test.ts` — bit-identical parity gate.
- `scripts/ecs-spike-bench.ts` — wall-clock OOP-vs-ECS benchmark.

### Simplifications (documented so the parity claim is honest)

- Player-as-target branch omitted (one fixed extra candidate, orthogonal to SoA).
- Cluster-aware target distribution omitted; plain nearest only (the brute-force
  branch is what is mirrored).
- State reduced to ALIVE/DEAD (production has 13 states; the targeting scan only
  reads the DEAD distinction).

Both paths apply the **same** simplifications and round through **Float32** (the
ECS columns are `Float32Array`; the OOP `Vec3F32` records are Float32-backed) so
parity is bit-for-bit, not "approximately equal".

## 3. Parity result

`src/ecs/EcsParity.test.ts` — **pass 10/10**. ECS target assignments are
element-for-element identical to the OOP reference across:

- Benchmark sizes 120/500/1000/2000 plus edge sizes 1/2/7/33 (seed 2718).
- N=1000 across 5 distinct fixed seeds (1, 12345, 99991, 1664525, 2026).
- A non-no-op sanity assertion (real targets assigned; dead entities never select).

A SoA rewrite is therefore provably behavior-preserving for this loop. (Parity is
necessary but not sufficient — it gates the speed question; it does not answer it.)

## 4. Benchmark

Reproduce:

```
npx tsx scripts/ecs-spike-bench.ts --iters 200 --warmup 50 --seed 2718
```

Runtime: Node 24, Windows 11, AMD Ryzen 7 3700X, single-threaded JS. Per case:
build once (excluded — steady-state per-frame scan cost is what dominates at
3,000 NPCs), warm both paths, then time `iters` calls each. Best-of is the most
stable signal under a noisy OS scheduler; mean reported alongside. Two runs below
(200/50 and 300/80) to show stability.

### Bounded / GC behavior

The brute-force O(N²) scan is **not bounded** in either model — wall-clock grows
~4x from 1000→2000 (5.3→22.5 ms OOP; 5.6→22.5 ms ECS), i.e. quadratically, as
expected from N². That unboundedness is **algorithmic, not a data-layout
artifact**, so SoA cannot fix it (a spatial grid / BVH can, and production
already has one — this spike deliberately mirrors the brute-force branch). On
GC: the SoA columns are pre-allocated typed arrays (steady, no per-call growth),
but the ECS path's per-call `query()` + `indexByEid` Map build *add* short-lived
allocations the OOP path doesn't have — arguably worse for frame-time tails, not
better. Neither path exhibits unbounded heap growth across iterations; both are
steady-state at a fixed N.

## 5. Verdict + rationale

**DEFER** — promote criteria not met:

| Decision-rule clause (REARCHITECTURE.md § E1) | Measured | Pass? |
|-----------------------------------------------|----------|-------|
| ECS ≥ 3x faster at 1000+ entities             | 1.00x–1.07x | NO |
| ECS path bounded (no unbounded GC growth)     | bounded per-N, but adds query/Map allocs; loop is O(N²) by algorithm | N/A — speed gate already failed |
| Port bounded (estimable < 2 weeks)            | combatant port ~2–4 weeks (E1 § 5) | NO |

Why the theoretical SoA cache advantage doesn't materialize here (consistent
with E1 § 4 "Interpretation"):

1. **V8 already optimizes stable-shape OOP access.** The `Vec3F32` records get
   hidden-class inline caching; field reads compile to direct offset loads. The
   "10-100x" SoA folk wisdom is a C/C++ figure that does not survive contact with
   modern V8.
2. **The loop is arithmetic-bound, not memory-bound.** The O(N²) inner loop is
   dominated by multiply/compare/branch on values already in registers/L1; the
   memory-access *pattern* barely moves wall-clock. AoS already streams its
   position records near-linearly, so SoA's locality edge is marginal.
3. **bitECS adds per-call overhead.** `query(world, [...])` assembles an array of
   eids each call; the spike also builds an `indexByEid` Map for range lookup +
   result placement. That overhead eats any raw SoA win — exactly the effect E1
   saw with its query-less "dense" sanity bench (~parity) vs its framework bench
   (slower).
4. **A storage-layout change cannot fix an algorithmic problem.** The real
   combat120 tail (p99 ~34ms) is `AIStateEngage.initiateSquadSuppression()`
   synchronous cover search (open P0 in `docs/BACKLOG.md`), not scattered-heap
   cache misses. ECS does not touch that.

Two independent subsystems (E1: projectile physics ~0.9x; this spike: combatant
targeting ~1.0–1.09x) now both land short of the bar. That convergence is the
strongest possible signal for DEFER without further spiking.

### Reversibility

The spike is fully reversible (self-contained `src/ecs/`, dev-only dependency,
off the production import graph). A *real* port would be high-cost and
low-reversibility (combatant `target` refs become eid lookups, ~3700 tests touch
`Combatant` objects, external systems key on `combatant.id`) — the inverse of
what the reversibility-weighted rubric rewards, with no measured upside.

## 6. Fence / additivity status

- `bitecs ^0.4.0` is in **devDependencies** (`package.json`) — the production
  bundle is unaffected.
- Nothing outside `src/ecs/` imports the spike modules or `bitecs` (grep-verified;
  the spike is reached only by `EcsParity.test.ts` and `scripts/ecs-spike-bench.ts`).
- `src/types/SystemInterfaces.ts` is **untouched**. No `[interface-change]`.
- No production behavior change, no flag flip.

## 7. What a real migration would entail IF promoted (it is not)

For the record, so a future revisit starts from a plan rather than a blank page.
A real combatant ECS migration is **not** one change — it is a per-component-
system fan-out, each landed behind its own parity test (the pattern this spike
demonstrates at the scale of one loop):

1. **Component columns** — promote each hot `Combatant` field group to a SoA
   column set (position, velocity, health, state, faction, squad, target,
   LOD, cover anchors, terrain-sample cache). The `ComponentRegistry` grows
   large; many per-tick queries.
2. **Reference rewrite** — `combatant.target` (object ref) → `targetEid` with
   lookup dances; squad membership and cluster handles likewise.
3. **External-system re-keying** — navmesh adapter, spatial grid, terrain
   sample cache, influence-map feedback are keyed by `combatant.id` today;
   each needs eid-keying or a dual-path bridge during transition.
4. **Per-system parity gates** — every ported subsystem (targeting, movement,
   LOS, state transitions) ships with a deterministic bit-identical parity test
   like `EcsParity.test.ts` *before* it flips, so divergence is caught at the
   seam, not in playtest.
5. **Test migration** — `src/integration/scenarios/` + combat-flow tests that
   construct `Combatant` objects directly need updating.

Estimated ~2–4 weeks for combatants alone (E1 § 5), ~6–10 weeks codebase-wide.
**Do not start this without a measured hot path where SoA enables vectorization
the OOP form cannot reach** — a much sharper question than "should we adopt ECS?"
and the only framing under which a future ECS spike should be reopened.
